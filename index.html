Introduction to ASIC Verification

ASIC Design Verification (DV) is the process of confirming that a hardware design, written in a Hardware Description Language (HDL) like Verilog or SystemVerilog, conforms to its architectural specification. In modern chip design, verification consumes up to 70% of the engineering time and resources, making it a critical phase in the semiconductor lifecycle.

Why is Verification Crucial?

Unlike software, which can be easily patched or updated after release, a physical chip (ASIC) cannot be changed once it is fabricated in a silicon foundry. A single logical bug can render the entire chip useless, costing millions of dollars in fabrication mask sets and delaying time-to-market. Therefore, functional verification ensures that bugs are caught before committing the design to silicon.

Key Stages of Verification
  • Simulation-based Verification: Creating a testbench that feeds stimulus to the design and monitors outputs.
  • Code Coverage: Measuring how much of the RTL code has been executed (line, branch, toggle, FSM coverage).
  • Functional Coverage: Measuring whether specific design features, scenarios, and protocols have been thoroughly tested.
  • Assertion-Based Verification (ABV): Using temporal logic (SVA) to continuously check protocol compliance at block interfaces.

Basics of Verilog HDL

Verilog is a standard Hardware Description Language (HDL) used to model electronic systems. It allows developers to specify digital hardware at various levels of abstraction, ranging from low-level gate-level structures to high-level behavioral algorithms.

Behavioral vs. Structural Modeling

In structural modeling, you instantiate logic gates and connect them with nets (wires). In behavioral modeling, you write sequential blocks (using always or initial blocks) to describe the logic. Modern digital designers primarily use behavioral coding because synthesis tools automatically compile it into optimized gate-level logic.

Blocking vs. Non-blocking Assignments

A common pitfall for beginners is understanding the difference between blocking (=) and non-blocking (<=) assignments. Blocking assignments execute sequentially within a block, resembling software. Non-blocking assignments execute in parallel, mimicking registers.

  • Use blocking assignments (=) for combinational logic blocks.
  • Use non-blocking assignments (<=) for sequential logic blocks triggered by a clock edge.

UVM Components Explained

The Universal Verification Methodology (UVM) is a structured SystemVerilog library designed to build reusable, clean, and highly configurable verification environments. It defines base classes and guidelines to streamline verification testbench architectures.

The UVM Testbench Hierarchy

A typical UVM testbench is structured hierarchically. The top-level class is the uvm_test, which instantiates the uvm_env. Inside the environment, there are agents, registers models, and scoreboards.

Core UVM Components
  • UVM Sequencer: Manages the transaction sequences and routes them to the driver.
  • UVM Driver: Receives transactions from the sequencer, translates them into pin-level signal toggles, and drives them to the Design Under Test (DUT).
  • UVM Monitor: Listens to the interface pins, captures the signals, reconstructs them into transactions, and publishes them.
  • UVM Scoreboard: Compares the observed transactions from the monitor against expected values (from a golden model) to check correctness.
  • UVM Agent: Packages a sequencer, driver, and monitor into a single reusable component representing an interface protocol.

Front-End VLSI Design Flow

The Front-End VLSI Design Flow covers all stages from system specification down to generating a verified gate-level netlist. This path focuses on logical functionality, timing verification, and preparing the circuit for the physical layout (back-end).

Step-by-Step Flow
  • Specification: Defining the system parameters, speed, area, power budget, and features.
  • RTL Design: Coding the hardware description in Verilog, SystemVerilog, or VHDL.
  • Functional Verification: Simulating the RTL code using testbenches to verify that the logical outputs match specs.
  • Lint Checking: Statically analyzing code to find potential issues like latch generation, uninitialized variables, or mismatched widths.
  • Logic Synthesis: Using tools like Synopsys Design Compiler to convert the behavioral RTL code into a library-specific gate-level netlist.
  • Static Timing Analysis (STA): Verifying that all paths meet setup and hold timing constraints before sending it to physical layout.

SystemVerilog Assertions (SVA)

SystemVerilog Assertions (SVA) are declarative statements used to verify the temporal behavior of a digital system. They act as active inline checkers that monitor signal protocols continuously during simulation, pinpointing the exact cycle a design failure occurs.

Immediate vs. Concurrent Assertions

Immediate assertions are checked at a specific evaluation point in sequential logic blocks, behaving like standard procedural checks in software. Concurrent assertions evaluate sequences of expressions over time, triggered by clock cycles, making them highly effective for protocol handshakes.

Anatomy of a Concurrent Assertion

Concurrent assertions typically use the sequence and property block structure. For example, a basic request-to-acknowledge handshake check: 

property p_handshake;
  @(posedge clk) disable iff (rst)
  req |-> ##[1:3] ack;
endproperty
assert property (p_handshake);

This assert property checks that whenever the request signal (req) goes high, the acknowledge signal (ack) must assert within 1 to 3 clock cycles. If it doesn't, a timing error is logged immediately.
